** Why
On [2026-05-15] one =20260515121426.8ea41ddd25714a7f@mg.ziprecruiter.com=
email caused the =dedup_scorer= agent to call =find_company_by_name("Visa")=
50 times against =mcp.careercaddy.online= before producing a (useless)
=DedupResult=. Triage row 1080, run 602; logfire traces start at
=2026-05-15T15:21:24Z= and span ~70s of gpt-4o-mini ↔ MCP ping-pong.

Root cause: the filter in =src/agents/agent_factory.py=
=_dedup_mcp_factory= allows =find_duplicate_candidates= — *a tool that
does not exist on the upstream public MCP server.* The actual tool is
=get_duplicate_candidates(job_post_id: int)=, which takes an existing
post id (wrong signature for a pre-POST check anyway). After the filter
strips the non-existent name, only =find_company_by_name= +
=search_companies= reach the agent. The system prompt
(=_DEDUP_PROMPT= in =src/agents/email_agents.py=) tells the model to
call =find_duplicate_candidates= — model can't find it, fixates on the
nearest-named tool, loops. No =UsageLimits= on the =agent.run= call
either, so it ran to gpt-4o-mini's default ~50-turn ceiling.

The dedup_scorer never had a working backend. The api-side dedupe-on-POST
(link match + content_fingerprint, with =merge_empty_fields_from_attrs=
on the existing row) already covers the link-bearing case. Inline
=link=NULL= posts lose the suffix-drift catch, but that's a smaller
hole than the cost+latency of a broken pre-POST scorer.
